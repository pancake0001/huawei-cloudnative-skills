# 最佳实践：使用OpenClaw配置、查询和治理CCE AOM告警

## 应用场景

CCE集群上线后，运维团队通常需要快速完成告警规则初始化，并在日常排障中查询当前活跃告警、历史告警、告警规则、通知规则和静默规则。通过OpenClaw Agent对接 `alarm-correlation-engine` Skill，可以实现：

- 一键为指定CCE集群创建AOM告警中心推荐规则
- 未指定通知规则时，自动创建集群级默认通知规则
- 按集群查询当前活跃告警和历史告警
- 对告警进行去重、分级和突发/常态分析
- 查询、停用、启用和删除告警规则
- 查询告警通知规则和静默规则
- 所有创建、修改、删除和启停动作均采用预览 + 确认机制

本实践以 **北京四 testxxx 集群批量配置CCE告警规则，并查询告警状态** 为例，演示客户如何通过自然语言使用工具完成告警配置和后续治理。

## 前提条件

- 已在OpenClaw Agent中注册 `alarm-correlation-engine` Skill。
- 已配置华为云访问凭证，建议通过环境变量或安全凭据文件注入，不在对话或文档中暴露AK/SK。
- 目标CCE集群已绑定AOM CCE Prometheus实例。
- 已准备可用的SMN主题，例如主题名 `test`。如果已有AOM通知规则，也可以直接复用。
- 执行账号具备AOM告警规则查询、创建、修改、删除、通知规则查询和通知规则创建权限。
- 创建类、修改类、删除类、启停类动作必须经过用户二次确认。

## 涉及的Skill

| Skill名称 | 功能说明 |
|-----------|---------|
| `alarm-correlation-engine` | AOM告警查询、活跃告警查询、告警归并分析、告警规则管理、通知规则和静默规则查询 |
| `huawei-cloud-cce-cluster-management` | 查询CCE集群列表，确认集群名称、集群ID和集群状态 |
| `observability-context-builder` | 在告警分析后继续汇聚指标、日志和K8s事件上下文 |

## 操作步骤

### 步骤1：向Agent描述告警配置需求

在OpenClaw对话中，输入自然语言需求：

```text
帮我给北京四集群testxxx批量创建CCE AOM告警规则，订阅主题使用test。
```

Agent会先识别目标区域和集群：

| 信息 | 示例值 |
|------|--------|
| 区域 | `cn-north-4` |
| 集群名称 | `testxxx` |
| 集群ID | `aa6e45a8-57de-11f1-a7f6-0255ac10026a` |
| 通知方式 | 使用集群级默认通知规则 |
| SMN主题 | `test` |

如果只提供集群名称，Agent会先查询CCE集群列表，找到对应集群ID后再继续。

### 步骤2：准备告警通知规则

批量创建规则前，需要确定告警通知方式。客户可以选择两种方式：

| 使用方式 | 客户输入示例 | 说明 |
|----------|--------------|------|
| 使用已有通知规则 | `通知规则使用已有的集群告警通知规则` | Agent会直接绑定指定通知规则 |
| 自动创建通知规则 | `订阅主题使用test` | Agent会为集群创建默认通知规则 |

需要查看已有通知规则时，可以直接让Agent查询：

```text
查询北京四的AOM告警通知规则。
```

如果未指定已有通知规则，客户需要提供SMN主题。可以提供主题名称，也可以提供完整主题URN。若缺少必要信息，Agent会提示补充。

### 步骤3：预览一键创建告警规则

创建前，Agent会先进行预览，不会立即创建规则。客户可以这样表达：

```text
预览为北京四集群testxxx创建CCE默认AOM告警规则，订阅主题使用test。
```

预览结果会说明：

- 将要创建的告警规则数量
- 指标告警和事件告警的数量
- 使用或自动创建的通知规则
- 是否需要用户确认
- 缺少哪些必要信息

默认规则集包括：

| 类型 | 数量 | 示例 |
|------|------|------|
| Prometheus指标告警 | 38 | Pod状态异常、Pod频繁重启、节点CPU使用率、节点磁盘可用率、节点Kubelet异常 |
| CCE事件告警 | 12 | Pod内存不足OOM、节点磁盘空间不足、节点状态异常、扩容节点超时、集群状态不可用 |

### 步骤4：用户确认后执行创建

确认预览无误后，客户明确回复：

```text
确认创建。
```

Agent再执行创建。执行完成后，会返回创建成功、跳过和失败的规则数量。

如果部分规则已存在，Agent会跳过同名规则，避免重复创建。

### 步骤5：验证告警规则落库

创建完成后，让Agent查询规则列表：

```text
查询北京四testxxx集群的告警规则。
```

Agent会按集群或规则名前缀过滤结果，确认规则数量：

| 校验项 | 期望结果 |
|--------|----------|
| `testxxx` 相关规则总数 | 50 |
| 指标告警 | 38 |
| 事件告警 | 12 |
| 创建失败 | 0 |

### 步骤6：查询当前活跃告警

当用户需要查看当前集群告警状态时，可以直接输入：

```text
获取北京四testxxx集群当前活跃告警。
```

输出应重点关注：

| 字段 | 说明 |
|------|------|
| `alarm_name` | 告警名称 |
| `alarm_status` | 当前状态 |
| `alarm_level` | 告警级别 |
| `resource_id` | 告警资源 |
| `starts_at` | 首次触发时间 |
| `description` | 告警描述 |

### 步骤7：查询历史告警并归并分析

当用户需要分析一段时间内的告警趋势时，可以输入：

```text
分析北京四testxxx集群最近的AOM告警，帮我区分突发告警和常态告警。
```

分析结果建议按以下结构输出：

| 分类 | 说明 | 处理建议 |
|------|------|----------|
| 突发告警 | 最近集中出现，可能与变更、流量或资源异常有关 | 继续调用根因分析或日志/事件分析Skill |
| 持续告警 | 长时间未恢复，通常需要排障或容量治理 | 优先处理高等级和业务相关资源 |
| 常态噪音 | 频繁出现但影响较低或已知误报 | 调整阈值、降噪或静默规则 |

### 步骤8：停用、启用或删除告警规则

客户可以通过自然语言管理规则：

```text
停用testxxx集群的节点磁盘空间不足告警规则。
```

```text
启用规则1510002824719630341。
```

```text
删除北京四testxxx集群相关的告警规则。
```

对于停用、启用和删除操作，Agent应先展示影响范围并等待用户确认。批量删除时，Agent需要先列出将删除的规则，确认仅包含目标集群规则后再执行。

## 预期结果

完成本实践后，OpenClaw Agent能够帮助用户完成以下闭环：

1. 识别目标区域、集群名称和集群ID。
2. 查询或自动创建集群级AOM通知规则。
3. 在缺少SMN主题时提示用户补充必要输入。
4. 预览批量告警规则创建计划。
5. 用户确认后创建CCE默认告警规则。
6. 查询并验证告警规则数量和类型。
7. 查询当前活跃告警、历史告警和归并分析结果。
8. 对告警规则进行受控停用、启用和删除。

## 约束与注意事项

- 可以不指定已有通知规则。未指定时，Agent会使用集群级默认通知规则。
- 自动创建通知规则时必须提供可用的SMN主题名称或主题URN。
- 指标告警依赖目标集群绑定AOM CCE Prometheus实例。
- 事件告警依赖CCE事件上报链路，建议同时确认日志采集和节点故障检测插件状态。
- 创建、修改、删除、停用、启用动作必须经过用户明确确认后执行。
- 不要把AK/SK、Token、证书、真实项目ID写入文档或对话输出。
- 批量删除时必须先查询规则列表并限定前缀或规则名白名单，避免误删其它集群规则。

## 常见问题

### 找不到Prometheus实例

错误示例：

```json
{
  "success": false,
  "error": "未找到目标集群对应的Prometheus实例"
}
```

处理建议：

1. 查询AOM Prometheus实例，确认目标集群是否绑定CCE类型实例。
2. 查询CCE插件，确认云原生监控相关组件是否已安装。
3. 若控制台刚开启监控，等待实例绑定信息同步后重试。

### 查询到的通知规则数量和控制台不一致

建议让Agent重新查询当前区域的AOM通知规则。如果仍不一致，应确认当前凭证、项目和区域是否与控制台一致。

### 自动创建通知规则失败

请优先确认SMN主题是否存在，以及当前账号是否具备AOM通知规则和SMN主题访问权限。确认后重新发起创建即可。

## 后续建议

- 规则创建完成后，建议立即查询一次目标集群的告警规则，确认规则数量、通知方式和启用状态符合预期。
- 新集群上线初期建议每天查看活跃告警和高频历史告警，观察是否存在阈值过严、重复通知或长期未恢复告警。
- 对确认无业务影响的高频告警，先分析触发对象和触发时段，再决定是否调整阈值、配置静默或优化通知范围。
- 生产集群建议定期导出告警规则清单，作为变更审计和故障复盘材料。
- 告警触发后，建议先查看告警归并分析结果，再继续查询相关日志、事件和指标，避免只根据单条告警做恢复动作。
