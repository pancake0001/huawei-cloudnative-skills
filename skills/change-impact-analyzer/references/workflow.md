# Workflow

# # 1. Scope & Ingestion

1. Collect `region`, `cluster_id`, `namespace` or the entire cluster scope, target object, fault phenomenon, `fault_time`, `start_time`/`end_time` or `hours`.
2. Generate an `Analysis-Trace-ID`, which must be retained in reports and structured output.
3. Collect four types of shadow data in parallel:
   - Application and configuration changes: Deployment/StatefulSet/DaemonSet, ConfigMap, and Secret write operations in CCE audit logs.
   - Network and routing changes: Service, Ingress, Gateway API, HTTPRoute/TCPRoute and other write operations.
   - Security policy changes: NetworkPolicy, Role, ClusterRole, RoleBinding, ClusterRoleBinding, ServiceAccount write operations.
   - Infrastructure changes: Node writes, taint/cordon/drain signs, node pool current snapshot, cluster plugin/core configuration snapshot.
4. Synchronously collect response signals: K8s historical events, AOM active+history alarms, and current Pod/Service/Ingress/Node/ConfigMap/Secret/Security Group/VPC ACL snapshots.

# # 2. Filtering & Categorization

1. Only write operations such as `create`, `update`, `patch`, `delete` and `replace` are retained.
2. Noise reduction:
   - Discard Lease, Event, TokenReview, SubjectAccessReview, Pod status, Endpoint/EndpointSlice regular controller writes.
   - Discard running status writing such as ServiceAccount token sub-resource creation, Node status patch, NPD/kubelet heartbeat, etc.
   - Discard the pod binding of kube-scheduler and the writes generated by the deployment/replicaset/statefulset/daemonset controller to advance the state; they are control plane closed loops, not user changes.
   - Discard all writes to the `/status` subresource; the audit log may echo the full spec, but the status subresource itself is not a configuration change.
   - Discard HPA or controller workload updates that modify only `replicas`.
   - Discard short-lived objects such as temporary token Secrets.
   - Discard CCE platform self-managed RBAC, such as `system:cce:*`, `cce:*` and the executor is maintenance update of CCE/platform components; RBAC changes for ordinary users or unknown objects are still retained as high risk.
3. Semantic field retention:
   - Workload: `image`, `env`, `resources`, `readinessProbe`, `livenessProbe`, `ports`, `selector`, `affinity`, `tolerations`.
   - Config: `data`, `stringData`, CoreDNS `Corefile`, upstream DNS.
   - Network: Service `ports/selector`, Ingress/Gateway `rules/tls/backend`.
   - Security: NetworkPolicy `ingress/egress/policyTypes`, RBAC `roleRef/subjects`.
   - Node: `taints`, `unschedulable`, scheduling related fields.

# # 3. Impact & Blast Radius Modeling

1. Service change: Use selector to find the current matching Pod, and check the Ingress that references the Service.
2. Ingress/Gateway changes: extract the backend Service and infer the path from the external entrance to the backend microservices.
3. Workload change: Associate the current Pod according to the workload name, and then check the Service selector.
4. ConfigMap/Secret changes: Ordinary objects are prompted according to the namespace impact; CoreDNS/kube-proxy/core plug-in configuration is processed according to the entire cluster.
5. Node changes: associate with the Pod on the node; taint/unschedulable associate with Pending, FailedScheduling, and Evicted events.
6. NetworkPolicy/RBAC changes: weighted by security boundary span; the current version first associates audit objects with event/alarm keywords, and then supplements the current policy topology query.

# # 4. Risk Scoring

Basic points:

| Change Category | Typical Core Operations | Scope of Impact | Basic Risks |
| --- | --- | --- | --- |
| Basic configuration | CoreDNS / kube-proxy / Core plug-in configuration | Global | Critical / 90 |
| Security Policy | NetworkPolicy / RBAC Privilege Escalation or Convergence | Cross-service or cross-namespace | High / 75 |
| Infrastructure | Node taint / cordon / drain / Node pool changes | Nodes and scheduling paths | High / 70 |
| Network routing | Ingress / Gateway / Service port or backend changes | Ingress to microservices | Medium / 60 |
| Configuration object | ConfigMap / Secret | Dependent objects in the namespace | Medium / 55 |
| Workload | Mirrors, probes, resources, environment variables | Single service and upstream and downstream | Low-Medium / 45 |

Bonus points:

- The closer to `fault_time`, add 0-15 points.
- If relevant K8s events appear in the window after the change, 0-10 points will be added.
- If relevant AOM alarms appear in the window after the change, 0-12 points will be added.
- The more entities the current topology affects, add 0-12 points.
- Hits on the user target object or target namespace add 4-8 points.

# # 5. Reporting

The report must contain:

1. Analysis summary and `Analysis-Trace-ID`.
2. Describe the troubleshooting process and explain what is done in the four stages.
3. Data source collection status and gaps.
4. Core change timeline.
5. Top N highest risk warning.
6. Explosion radius and propagation path.
7. Evidence Matrix.
8. Conclusion, confidence level, and next step read-only verification recommendations.
